# ingest 技能

## 核心工作流：Inbox → Wiki → Delete

你正在维护一个 **LLM Wiki**（Obsidian 知识库）。`raw/` 目录是"待处理收件箱"，`wiki/` 是"编译输出层"。源文件经 ingest 编译为详细 wiki 页面后即完成使命，直接删除。

**目录结构约定：**
- `raw/01-articles/` — 网页剪藏的 Markdown 文章
- `raw/02-papers/` — 论文和 PDF 文献
- `raw/03-transcripts/` — 视频转录文案
- `raw/04-meeting_notes/` — 头脑风暴或会议纪要
- `wiki/sources/` — 资料摘要
- `wiki/entities/` — 实体（人物、公司、工具、产品）
- `wiki/concepts/` — 概念（框架、方法论、理论）

## 触发逻辑

1. **用户执行 `/ingest`**：扫描 `raw/` 下 `01-articles/`、`02-papers/`、`03-transcripts/`、`04-meeting_notes/` 四个目录，找出待处理文件。
2. **用户执行 `/ingest <path>`**：仅处理指定文件。
3. **隐式触发**：用户说"把这个资料摄入知识库"、"导入这篇文章"时，自动执行 ingest。

## 编译流水线

对每个待处理源文件，严格按以下步骤执行：

### 步骤 0：定位待处理文件

- **扫描范围**：仅扫描 `raw/01-articles/`、`raw/02-papers/`、`raw/03-transcripts/`、`raw/04-meeting_notes/` 四个目录
- **Glob 示例**：使用 `raw/01-articles/**/*` 等按目录逐个扫描

### 步骤 1：读取源文件

- **如果是 `.md` 文件**：使用读取工具完整读取内容。
- **如果是 `.pdf` 文件**：使用读取工具尝试提取文本。如果无法提取或内容为空，改为记录文件元信息（文件名、页数）在 sources 页面中。

### 步骤 2：提炼核心并扫描关联摘要

从源文件中提取：
- **核心主旨**：这段资料讲什么（1-2句话）
- **实体**：人物、公司、工具、产品等具体名词
- **概念**：框架、方法论、理论等抽象名词
- **所有可操作的细节**：命令示例、配置步骤、参数说明、常见错误及解决方案、故障排查对照表

如果是非中文内容，则翻译成中文。

**同时扫描 `wiki/sources/` 和 `wiki/index.md`**，找出与新内容主题相关的已有摘要页面。判断标准：
- 涉及相同实体或概念
- 属于同一领域/话题
- 补充或深化已有摘要的内容

### 步骤 2.5：合并决策 ⛔ 必须询问用户

**此步骤不可跳过。** 根据步骤 2 的扫描结果，使用 AskUserQuestion 工具向用户提问：

**如果找到 1 个以上相关摘要：**
> 发现以下与新内容相关的已有摘要页面：
> - [[摘要-xxx]] — 一句话描述
> - [[摘要-yyy]] — 一句话描述
>
> 是合并到已有摘要，还是新建独立摘要？

选项：
- "合并到 [[摘要-xxx]]"（每个相关摘要一个选项）
- "新建独立摘要"

**如果未找到相关摘要（全新主题）：**
跳过此步骤，直接进入步骤 3 创建新摘要。

### 步骤 3：创建或更新来源摘要

**模式 A — 创建新摘要**（用户选择"新建"或无相关摘要）：

在 `wiki/sources/` 创建 Markdown 文件：

```markdown
---
title: "摘要-文件slug"
type: source
tags: [来源, 原始文件]
sources: [raw/0x-xxx/xxx.md]
last_updated: YYYY-MM-DD
---

## 核心摘要
[3-5句话的核心总结]

## 关键要点
- [要点1]
- [要点2]
- ...

## 关联连接
- [[EntityName]] — 关联实体
- [[ConceptName]] — 关联概念
```

文件名使用 kebab-case：`摘要-{文件slug}.md`

**模式 B — 合并到已有摘要**（用户选择了某个已有摘要）：

1. 读取用户选择的已有摘要页面
2. **增量更新**该摘要页面：
   - 在 `## 核心摘要` 末尾追加新内容（保留旧内容，新增段落以 `> 新增 (YYYY-MM-DD):` 开头标注来源）
   - 在 `## 关键要点` 末尾追加新要点
   - 合并 `tags`（去重）
   - 更新 `last_updated` 日期
3. 将新源文件路径添加到该摘要的 frontmatter `sources` 列表中
4. **关联连接**如有新增实体/概念则追加

### 步骤 4：知识网络化（实体/概念页面）—— ⛔ 必须询问用户

对于步骤 2 提取的每个实体和概念，**不可自动创建新页面，必须先询问用户**。

**目标目录：**
- 实体 → `wiki/entities/`
- 概念 → `wiki/concepts/`

#### 步骤 4.0：创建决策（必须询问）

整理出所有提取到的实体/概念，按以下分类列出：

- **已存在页面**：列出 `wiki/entities/` 或 `wiki/concepts/` 中已存在的同名/同主题页面
- **不存在页面**：列出当前 wiki 中尚无对应页面的实体/概念

使用 AskUserQuestion 工具向用户提问：

> 从源文件中检测到以下实体/概念：
> - [[Git]] — 已存在页面
> - [[GitHub]] — 已存在页面
> - SSH — 尚未创建页面
>
> 是否需要创建新页面或更新已有页面？

选项：
- **创建新页面**：仅为不存在的实体/概念创建新页面（已有页面保持不变）
- **填充已有页面**：仅对已有页面进行内容填充，不创建任何新页面
- **全部处理**：创建新页面 + 填充已有页面
- **跳过**：不处理任何实体/概念页面

**如果用户选择"填充已有页面"或"全部处理"：**

对每一个**已存在**的实体/概念页面执行：
1. 读取现有页面完整内容
2. **保持原内容结构不变**，将新信息以"填充"方式补充进去：
   - 在已有区块末尾追加新命令/示例（用 `# 新增 (YYYY-MM-DD)` 标注）
   - 补充缺失的故障排查条目
   - 扩展关键操作速查表
   - 在 `## 关联连接` 中追加新来源
   - 更新 `last_updated` 日期，追加新 `sources`
3. **绝不删除或重写**已有区块
4. **发现冲突** → **立即暂停**，向用户报告冲突内容，询问处理方式后再继续

**如果用户选择"创建新页面"或"全部处理"：**

对每一个**不存在**的实体/概念页面执行：
1. 按照下方模板创建新页面，**必须达到详细程度标准**
2. 在 `sources` 中记录来源

**如果用户选择"跳过"：**

直接进入步骤 5，不创建也不更新任何实体/概念页面。

### 步骤 5：更新全局注册表

**更新 `wiki/index.md`：**

- Sources: `[[摘要-source-slug]] — 该资料的核心主旨`

- Concepts: `[[ConceptName]] — 该概念或框架的核心定义`

> 注意：log.md 位于项目根目录，日志内容使用纯文本（禁止 wikilink），避免在 Obsidian 知识图谱中制造无关连接。

### 步骤 6：删除源文件 ⛔ 强制执行

**此步骤不可跳过。** Wiki entities 已达到并超越源文件的详细程度，保留 raw 仅造成内容冗余。在确认以下全部完成后，**必须立即删除**源文件：

- sources 页面已创建
- 实体/概念页面已创建或更新，且达到详细程度标准
- index.md 已更新
- log.md 已更新

**删除命令：**
```bash
rm "<源文件路径>"
```

**删除后验证：**
确认 `raw/01-articles/`、`raw/02-papers/`、`raw/03-transcripts/`、`raw/04-meeting_notes/` 四个目录在处理完毕后均为空（或只剩 `.gitkeep` 等占位文件）。

## Obsidian 排版规范 ⛔ 强制执行

所有 entity 和 concept 页面必须遵循以下排版规则，确保在 Obsidian 中主次分明、阅读体验一致：

### 1. 顶部导航地图（必须）

每个 entity/concept 页面在标题下方、正文之前必须包含 `## 导航地图`，使用 Obsidian 页内 wikilink `[[#标题]]` 格式：

```markdown
## 导航地图

点击即可跳转到对应区块：

- 🚀 [[#安装与配置|安装与配置]] — 一句话说明这个区块是什么
- ⚙️ [[#核心机制]] — 一句话说明
```

- 使用 emoji 图标区分不同类型区块
- 每个链接后加 `— 一句话说明` 帮助读者判断是否需要跳转
- 导航地图前后用 `---` 分隔线与正文隔开

### 2. Obsidian Callout（必须）

关键提示必须用原生 Callout，禁止用普通 `>` 引用块：

| 场景 | Callout 类型 | 颜色 |
|------|-------------|------|
| 警告、踩坑、不可逆操作 | `> [!warning]` | 橙色 |
| 危险、数据丢失风险 | `> [!danger]` | 红色 |
| 小技巧、快捷方式 | `> [!tip]` | 绿色 |
| 补充说明、背景信息 | `> [!info]` | 蓝色 |
| 摘要、总结 | `> [!abstract]` | 灰色 |

```markdown
> [!warning] 关于凭据管理器
> 没有它的话，每次 `git push` 都会卡在身份认证环节。

> [!tip] 绑定过上下游后
> 执行过 `git push -u origin main` 后，后续只需 `git push` / `git pull`。
```

### 3. 大区块分隔

每个二级标题（`##`）大区块之间用 `---` 水平线分隔，让滚动时视觉边界清晰。

### 4. Entity 合并与薄重定向

当多个实体高度相关、内容大量重复时（如 Git 与 GitHub）：

1. **以内容最完整的页面为主干**（如 `Git.md`）
2. **将其他实体的独有内容并入主干**，用独立子章节收纳（如 `## GitHub 平台操作`）
3. **被合并的实体改为薄重定向页面**：只保留一句话定义 + 指向主干页面的链接
4. **保留双链可用**：`[[被合并实体]]` 链接仍然有效，点击后看到重定向提示

### 5. Source 归并

主题相似的 source 页面应合并：
- 通用指南保留为主体 source
- 具体案例/记录的内容拆分：通用部分填充到 entity，案例部分并入主体 source 的"实操案例"区块
- 删除被归并的独立 source 页面
- 更新 `index.md` 移除已删除页面

## 详细程度标准 ⛔ 强制执行

Wiki 页面的详细程度**必须以 raw 原始文件的细节密度为参照标准**，不可丢失以下信息：

### 实体页面的最低标准

| 必须有 | 说明 |
|--------|------|
| 完整安装/配置流程 | 不只是列命令名，要有具体命令、参数解释、为什么这样做 |
| 带注释的代码示例 | 每条命令上方或行尾用 `#` 注释说明用途，不只是贴代码块 |
| 关键注意事项/踩坑点 | 源文件里的 `> 关于 xxx` 警告块必须原样保留进 entity |
| 故障排查表 | 如果源文件有"常见问题"或"症状→原因→解决"，必须完整保留到 entity |
| 操作速查表 | 高频命令必须以表格形式呈现，不只是散落文中 |

### 实体页面判断口诀

> 如果用户在 entity 页面找不到源文件里已有的某条命令解释、某个踩坑提示，那就是详细程度不达标。

### 反例：不合格的实体页面

```markdown
## 关键信息
- 初始化仓库：git init 创建 .git 文件夹
- 远程协作：git push / git pull 实现同步
- 分支管理：git branch -M main 可重命名分支
```

问题：只有干条目，没有命令上下文、没有参数解释、没有踩坑提示。raw 里的 `credential.helper` 说明、Token 认证说明、`git rm` 删除流程全丢了。

### 正例：合格的实体页面

参照当前 `wiki/entities/Git.md`、`wiki/entities/PPT-Master.md` 的详细程度——每个区块都展开到可独立操作的程度。

## 完成前自检清单

每轮 ingest 结束前，逐项确认：

- [ ] 步骤 2.5 合并决策已询问用户（有相关摘要时）
- [ ] 步骤 4.0 实体/概念创建决策已询问用户
- [ ] 源文件已从 `raw/0x-*/` 中删除
- [ ] `wiki/index.md` 已更新且链接正确（新增页面时）
- [ ] `log.md` 已追加操作记录
- [ ] 所有页面（含合并更新的摘要）包含 `## 关联连接` 区域（无孤岛页面）
- [ ] `raw/01-articles/`、`raw/02-papers/`、`raw/03-transcripts/`、`raw/04-meeting_notes/` 四个目录中没有残留的已处理文件
- [ ] **Obsidian 排版规范达标**：导航地图、Callout、分隔线、锚点链接
- [ ] **实体页面详细程度达标**：命令带注释、踩坑保留、故障排查表完整
- [ ] **合并模式**：关联的实体/概念页面已同步刷新
- [ ] **Source 归并检查**：主题相似的 source 已合并，无冗余独立页面

## 冲突处理流程

当发现新旧知识冲突时：

1. **暂停**：停止当前 ingest 流程
2. **报告**：向用户说明冲突内容（哪个页面、冲突点是什么）
3. **询问**：请用户选择处理方式：
   - A) 保留新旧两者，标注为"知识冲突"
   - B) 用新知识覆盖旧知识
   - C) 放弃本次 ingest
4. **继续**：根据用户选择继续或终止

## 注意事项

- 所有 wiki 页面必须包含 `## 关联连接` 区域，不能产生孤岛页面
- 使用简体中文编写所有内容
- 实体命名使用 TitleCase，概念和来源使用 kebab-case
- 删除源文件是 ingest 流程的**必要组成部分**，源文件留在收件箱即为 ingest 未完成
- **实体页面的详细程度以源文件为基准，不可降级简化**
